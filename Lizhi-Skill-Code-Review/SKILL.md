# Lizhi-Skill-Code-Review：Skill 文件的代码审查与重构引擎

> 给 Skill 本身做 Code Review与优化。

一个 SKILL.md 就是给 LLM 执行的「代码」。被反复修改后，它会像任何代码一样腐化：补丁压补丁、规则互相打架、删过的功能留下孤儿引用、同一个概念有好几个名字。这个技能做的事，等同于对代码做**保持行为不变的重构（behavior-preserving refactor）**——只降低熵，不改变它实际怎么跑。

## 一条铁律：先锁行为，再动文件

重构最大的风险，是把一条**看着像废话、其实在兜底**的指令优化掉。

Skill 里的防御式指令往往是踩坑换来的教训。比如「禁止用 Write 工具写 MD，开销高 200 倍」——读起来像啰嗦，实则承载真实失败模式。**删掉它，行为就变了。**

所以顺序不能反：

1. **先把"这个 Skill 当前实际怎么跑"完整提取出来，形成行为契约。**
2. 只重写"表达"，不动"行为"。
3. 改完后重新提取一次契约，和原契约逐条对照——必须完全一致。
4. 任何会改变契约的改动，都不算"清理"，必须单独拎出来当"设计改进"请用户拍板。

**判不准某条约束是否必要时，保留它，标注"存疑·建议保留"，绝不擅自删除。** 宁可留一条冗余，不可丢一条兜底。

## 另一条原则：规则 / 配置 / 记忆分离

SKILL.md 里只该留**通用规则**——一套换谁来用、换什么场景都成立的执行逻辑。一旦某段内容掺进了"具体某个人的信息或偏好"，它就不属于规则文件，应该分离出去。分三层：

| 层 | 放什么 | 判据 | 例子 |
|----|--------|------|------|
| **规则**（留在 SKILL.md） | 通用执行逻辑，与具体用户无关 | 换个人用，这条仍然成立 | 「评审分对不对/值不值两个角度」 |
| **配置**（分离到配置文件） | 与本 Skill 强相关、但因人/场景而异的信息与倾向 | 只在这个 Skill 里用得到，换个用户取值会变 | 某人惯用的输出语气、默认项目、特定路径偏好 |
| **记忆**（交给 agent 记忆） | 跨 Skill 通用的用户信息与偏好 | 别的 Skill 也会用到同一条 | 用户名、母语、长期工作习惯 |

**怎么判**：问两个问题——① 这条内容换个用户来用还成立吗？不成立 → 不是规则。② 它只服务这一个 Skill，还是哪个 Skill 都用得上？只服务本 Skill → 配置；跨 Skill → 记忆。

审查时发现规则文件里夹带了应属配置或记忆的内容，标出来、建议分离，但**分离同样要保行为不变**——内容是搬家不是删除，搬走后规则文件要留一个读取入口（从配置/记忆取值），否则就改变了行为，需降级为待确认的设计改进。

## 工作流

### 第 1 步：读全

- 读 `SKILL.md` 全文。
- 读它引用的**每一个**附属文件（`scripts/`、`references/` 等）。建一张引用关系图：哪些文件被提到、被谁提到、是否真实存在。
- 没读到引用的文件就如实说，不靠猜测推断它的内容。

### 第 2 步：提取行为契约（重构的基准线）

把这个 Skill「可观测的行为」逐条抽出来，形成一张 checklist。这是后面判断"行为有没有变"的唯一标尺：

| 契约项 | 抽取什么 |
|--------|---------|
| 触发条件 | 什么情况下这个 Skill 会被调用（来自 description + 正文） |
| 输入 | 接受什么材料、怎么获取 |
| 执行步骤 | 实际会做哪几步、顺序如何、有无分支 |
| 硬约束 | 所有"必须 / 禁止 / 只能"级别的规则，逐条记下 |
| 输出 | 产出什么、结构如何、写到哪里 |
| 副作用 | 调用的工具、脚本、写文件、发消息等外部动作 |

### 第 3 步：审查（找腐化）

对照下面八类问题逐项扫描，每发现一处记下「位置 + 证据原文 + 类型」：

1. **矛盾**：两条规则互相打架（A 处说必做、B 处说禁止）；或同一件事在两处描述已经分叉、对不上。
2. **冗余**：同一条指令在多处重复；几段话说的是同一件事。
3. **失效引用**：提到的文件 / 脚本 / 章节不存在；为某个已删除功能保留的孤儿规则；改过名后残留的旧名字。
4. **术语漂移**：同一个概念有多个叫法；名称（如 Skill 名、模块名）在文件里不统一。
5. **补丁味**：防御式过度规定，「禁止 X、禁止 Y、禁止 Z」越堆越长；括号里的补充说明层层叠加；特例越打越多。
6. **结构腐化**：章节顺序混乱；描述的输出结构和实际输出模板对不上；示例和现行规则矛盾；存在永远走不到的死分支。
7. **过程性残留**：把"制作这个 Skill 时的讨论/演化痕迹"写进了成品。比如「本技能只读不写，故仅保留读取侧」「这是精简版，完整版见 X」「改名后这里统一为 Y」——这些是开发过程的旁白，对运行毫无价值，只会干扰理解。成品只该陈述"现在是什么、怎么用"，不该解释"为什么变成这样"。一律删除（承载行为的"禁止 X 因为会 Y"型教训不算，那是兜底，保留）。
8. **可移植性 / 隐私泄露**：硬编码了只在作者机器上成立的信息——绝对路径（如 `/Users/某人/...`）、个人用户名、内网地址、密钥、机器专属配置。这类内容既泄露作者隐私，又让 Skill 到别人手里直接失效。改为可移植写法：相对本 Skill 目录的路径、环境变量、PATH 查找、运行时占位符（如 `<SKILL_DIR>`）。
9. **层次错放（规则/配置/记忆）**：规则文件里夹带了本该分离出去的内容——因人而异的偏好/信息（应进配置），或跨 Skill 通用的用户信息（应进记忆）。判据见上文「规则 / 配置 / 记忆分离」。发现后建议分离，但搬走后要留读取入口，保行为不变。

### 第 4 步：出诊断 + 重写方案（先不动文件）

按「输出结构」给出完整方案，**包含一张契约对照表证明行为不变**。等用户确认。

### 第 5 步：改写（确认后）

改写遵循「先全量备份 → 再应用 → 后验证」，任一环节不通过都能一键回到原状。

- **备份（动手前的第一件事）**：把待改 Skill 的**整个目录**快照到 `.skill-review-backup/<时间戳>/` 下，而不只备份 `SKILL.md`——因为重构可能同时动到 `scripts/`、`references/` 等附属文件，单文件 `.bak` 兜不住。备份后向用户报告快照路径。
- **应用重写**：逐文件应用第 4 步已确认的方案。
- **验证**：重新执行第 2 步，从新文件提取契约，和原契约逐条比对。
- **判定与回滚**：
  - 完全一致 → 通过，保留备份待用户确认后清理。
  - 有差异且属于"已批准的设计改进" → 通过，在报告中标明这条差异是预期的。
  - 出现"未预期的差异"（即引入了 bug）→ **立即从最近一次备份整目录回滚，恢复原状**，不做手工补丁式修补，然后说明哪一步出错、重走第 4 步。

### 备份与回滚约定

- **备份粒度**：整个 Skill 目录，不是单文件。一次审查会话只创建一个时间戳快照，多文件改动同属一次快照，保证能一致地整体回滚。
- **回滚触发**（满足任一即回滚）：契约出现未预期差异；用户在确认前喊停；改写中途失败导致文件处于半完成状态。
- **回滚动作**：用对应时间戳目录覆盖回原位置，使 Skill 回到改写前的完整状态；回滚后复跑第 2 步契约提取，确认已还原。
- **备份生命周期**：备份默认保留，**绝不在用户确认结果前删除**。用户确认改写无误后，方可按其指示清理 `.skill-review-backup/`。
- **破坏性操作**：覆盖、删除备份等操作执行前向用户说明，不静默进行。

## 重构手法

发现问题后，用这些手法修——目标始终是「更少的字，承载一样多的行为」：

- **单一信息源**：每条规则只在最相关的一处讲清楚，别处引用，不重复正文。
- **合并同类**：散落各处、讲同一件事的规则，归拢到一处。
- **化补丁为规则**：把堆叠的「禁止 X、禁止 Y」反向提炼成一条干净的正向规则（「只用 A 方式」往往比「禁止 B、禁止 C、禁止 D」更短也更稳）。
- **删死代码**：移除指向不存在文件 / 功能的引用和孤儿规则——但删之前必须确认它真的没用，而不只是看着没用。
- **删过程旁白**：删掉解释"这个 Skill 怎么演化成现在这样"的话（精简版/完整版对比、改名记录、"故仅保留 X"），只留"现在是什么、怎么用"。
- **去机器绑定**：把绝对路径、用户名、内网地址、密钥等只在作者机器成立的信息，换成相对路径 / 环境变量 / PATH 查找 / `<SKILL_DIR>` 占位符，让 Skill 可移植、不泄隐私。
- **分层归位**：把规则文件里夹带的因人而异内容搬到配置、跨 Skill 内容搬到记忆，原处留一个读取入口，保持行为不变。
- **统一术语**：一个概念一个名字，全文替换到底。
- **保形精简**：长句拆短、去掉口水词、统一格式，但不动任何承载行为的词（"必须""禁止"及其宾语）。

**手法的红线**：以上每一步都不能改变第 2 步提取的任何契约项。一旦某个改动会动到契约，它就不是重构，而是"设计改进"，归到第 5 块单独处理。

## 输出结构（结论先行）

### ① 总体评估

```
一句话：这个 Skill 的健康度如何、最该先处理的一个问题是什么。
```

### ② 行为契约快照

第 2 步提取的 checklist。这是"重写后不许变"的基准，也让用户一眼看清这个 Skill 到底在做什么。

### ③ 问题清单

| 类型 | 位置 | 证据（原文片段） | 处理手法 |
|------|------|----------------|---------|
| 矛盾/冗余/失效引用/漂移/补丁味/结构/过程残留/可移植性/层次错放 | 章节或行 | 引用原文 | 对应的重构手法 |

判不准是否该删的，单独标「存疑·建议保留」，并说明顾虑。

### ④ 重写方案 + 契约对照

给出重写后的内容（小改用 diff，大改给完整新版）。紧跟一张对照表，逐条证明行为未变：

| 契约项 | 重写前 | 重写后 | 是否一致 |
|--------|--------|--------|---------|
| … | … | … | ✅ 一致 |

全部 ✅ 才是合格的"纯重构"。

### ⑤ 待确认的设计改进（如有）

会改变行为、但确实让设计更合理的改动，单列在此，**默认不执行**，逐条说明「改什么、为什么更好、会带来什么行为变化」，请用户逐条拍板。不和纯重构混在一起。

## 怎么说话

- **结论先行**：先给总体评估，再展开。
- **证据说话**：每条问题都引用原文片段，不空泛说"这里乱"。
- **保守优先**：拿不准就保留并标存疑。删除一条规则的举证责任，永远在"删"这一方。
- **重构与改进分开**：纯表达清理（行为不变）可以放手做；任何动到行为的，一律降级为待确认改进。
- **改完即验**：以契约对照为准，不靠感觉判断"应该没问题"。
