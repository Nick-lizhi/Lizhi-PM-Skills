# PM-Internal-Review-Pro：产品方案内审引擎

> 由 Nick-Lizhi 设计

输入一份产品方案，输出从理解到审定的完整判断。

## 设计哲学

这个 Skill 的底层只有一个问题：**这份方案值不值得做，以什么规模推进？**

所有判断动作最终都被四条线撑着：

- **假设链**——主张的前提站不站得住
- **差距感知**——问题定义与现实之间是否自洽
- **行动代价**——这个决策可逆吗，判断错了代价多高
- **传递噪声**——信号从现实到材料到你，损耗了多少

## 获取材料

- **POPO 文档**（`docs.popo.netease.com` 域名）-> **必须** 用 `popo_io.py` Python API 读取，**禁止** 用 WebFetch 或 `python3 popo_io.py url` 的 CLI 方式：
  ```python
  # popo_io.py 位于本 Skill 的 scripts/ 子目录，用 importlib 按绝对路径加载
  import importlib.util
  spec = importlib.util.spec_from_file_location(
      "popo_io", "<SKILL_DIR>/scripts/popo_io.py"

  text, doc_id = pio.read_doc_from_url("https://docs.popo.netease.com/lingxi/xxx")
  ```
  其中 `<SKILL_DIR>` 为本 Skill 所在目录的绝对路径，Claude 运行时根据 SKILL.md 位置自行替换。
- 普通网页 URL -> WebFetch 获取内容，**记录原始 URL 供后续写入文件元信息**
- 本地文件 -> Read，记录文件路径

获取后检查材料是否完整可读。不完整时告知用户，不强行推进。

## 分析指引

**通读全文一遍，在同一遍阅读中同时收集所有维度的信息。分析是一次性完成的整体判断，不按输出模块顺序分阶段进行。** 收集完毕后，按输出结构一次性写出所有内容。

以下是各维度的分析方法，阅读时并行积累，不需要逐个独立执行。

### 框架速查

| 框架 | 属于哪条线 | 核心问法 |
|------|-----------|---------|
| 假设树 | 假设链 | 这个主张成立需要哪些前提为真？ |
| 最少驱动因子 | 假设链 | 列出的前提里，哪些是真正独立的？去掉冗余，只保留最小集。 |
| SCQA | 差距感知 | 现状/冲突/核心问题/答案是否自洽？ |
| Type 1/2 | 行动代价 | 这个决策可逆吗？审查要花多少力气？ |
| Pre-mortem | 行动代价 | 假设失败了，最可能的原因是什么？ |
| Goodhart | 传递噪声（指标层）| 这个指标一旦成为 KPI，最容易的作弊路径是什么？会损害真正目标吗？|
| 金字塔原理 | 传递噪声（表达层）| 结论前置了吗？论据 MECE 吗？每条论点 So What？ |

注意：Lead/Lag 不适用于方案审定——方案尚未执行，没有可观测的执行信号。

**Goodhart 适用于方案评审**：方案尚未执行，但方案选定的成功指标本身可能存在 Goodhart 风险——当指标成为目标时，执行侧会优化指标而非优化真正的目标。在指标选择审查阶段（一致性检查：「背景认为重要的问题，指标体系中是否有对应度量？」）主动触发：这个指标一旦成为 KPI，最容易的作弊路径是什么？该路径会损害真正的业务目标吗？

### 诊断标注规范

**所有带有判断性质的结论，必须用「诊断」格式标注，不允许把判断混在叙述段落里。**

格式：
```
**诊断** 🟢 / 🟡 / 🔴  **[结论词]** — [一句话说明]
[展开解释，可多行]
```

灯色规则（严格遵守，不模糊套用）：

| 灯色 | 含义 | 典型触发场景 |
|------|------|------------|
| 🟢 | 成立 / 通过 / 一致 | 假设有充分支撑、数据自洽、框架自洽、必要性成立 |
| 🟡 | 部分成立 / 有疑点但不致命 | 假设证据弱但非空、解读未封闭、症状层干预、参数缺依据但量级合理 |
| 🔴 | 不成立 / 严重问题 | 数据内部不自洽、论证链断裂、假设无任何依据、缺失项影响决策、缺少回退预案 |

**诊断出现位置（穷举）**：
- 问题定义 → SCQA 自洽性
- 解法充分性 → 每个前提的可信度（战略层 + 执行层）
- 解法必要性 → 干预点对齐结论、替代方案覆盖度
- 关键断言校验 → 每条断言（含隐性断言、Goodhart 审查断言）的解读封闭性；跨断言矛盾在相关断言处双向标注
- 收益价值分析 → 收益规模合理性、增长路径薄弱节点
- 风险评估 → 每条失败路径的预案完整度
- 评审结论 → 总体判断

### 方案速览维度

阅读时提取三样东西，作为后续所有分析的基础。

**核心主张**：一句话识别——作者最终希望你做什么判断或决定？显式写出来，不要模糊化。

**骨架地图**：提取文档的逻辑层级，用带树形连接符的 ASCII 树结构呈现，将段落标记为三类：
- `[骨]` 承载核心论证的段落
- `[肌]` 展开论证的证据、数据、例子
- `[筋]` 过渡、连接

树形连接符规则（强制使用，不允许用纯缩进替代）：
- 中间子项用 `├──`
- 最后一个子项用 `└──`
- 父节点延续线用 `│   `
- 叶节点内容直接跟在连接符后，无需额外缩进符号

**概念扫清**：扫描材料中出现的专有名词、行业缩写、内部术语、数据指标名称，逐条简洁释义。目的不是翻译，而是让后续分析不被「看上去懂但未必准确」的词遮住。遇到无法判断含义的术语，标注「需用户补充」，不自行猜测。

### 思路分析维度

**问题定义**：用 SCQA（S/C/Q 部分）——问题框架是否自洽？冲突是真实存在的吗？问题定义不准，后续所有分析失效，在此停下并标注。

**解法充分性**：用假设树，分两层展开——两层严格分离，凡是需要知道解法具体设计才能判断的假设，归执行层；不需要的，归战略层：

- **战略层**（假设解法可执行，问这件事值不值得做）：价值主张成立，依赖哪些关于用户/市场/竞争的前提？用最少驱动因子去掉派生假设，只留最小独立集，哪个最脆弱？
- **执行层**（假设问题诊断正确，问解法能不能交付）：解法成立需要哪些结构性、逻辑性假设？哪个最容易断？**仅追逻辑链，不重复校验具体参数数字**——方案中的数字和参数统一交给「关键断言校验」处理。
- **多方案对比触发条件**：如果材料中出现方案 A/B/C 等多个选项，强制触发对比分析。检查材料是否已给出各方案的利弊对比；若未给出，标注为「缺失项」并生成对比框架（核心假设 / 预期收益 / 主要风险 / 资源成本 / 可逆性）。没有利弊权衡的多方案呈现，不构成有效的决策支撑，在评审结论中显式指出。

**解法必要性**：回答「这个解法是否在正确的干预点上介入」，三步实证结构：

① **根因溯源**：对问题定义中识别的核心冲突，纵向逐层下钻，直到触及不可再分的底层元素。

下钻规则：
- **纵向，不横向**：每次下钻只回答「为什么是这样」，不是「还有什么其他因素」
- **每层命名**：给每一层一个精准标签（2-4字），概括这一层看到的机理
- **层间找裂缝**：每层结尾识别出一个未被解释的矛盾或张力，那就是进入下一层的入口
- **到底的判断**：当再往下问「为什么」，答案只能是人性的基本结构、系统的物理约束、逻辑本身，或同义反复时，即已触底

下钻路径通常从「现象层」→「行为层」→「动机层」→「结构层」→「约束层」，具体层数因问题深度而定（3-6层），不强制套用固定层数。

② **干预点对齐**：这个解法是在根因层介入，还是在症状层介入？
   - 根因层介入 → 必要性成立
   - 症状层介入 → 解法有效果但问题会复发，在结论中显式标注「症状层干预」
③ **替代方案扫描**：同一根因是否存在更轻量/更快/成本更低的干预方式？
   - 材料已有多方案对比 → 检查对比框架是否基于根因对齐，而非资源/惯性驱动
   - 材料只有一个方案 → 强制生成 2-3 个替代路径，说明为何未选；若材料未给出理由，标注「缺失项」

### 事实分析维度

**关键断言校验**：自动提取材料中所有「需要为真，主张才能成立」的数字、假设、判断。**断言的范围包括隐性断言**——凡是材料在某处承诺了、暗示了、依赖了某个条件为真，即便没有显式陈述为数字，也构成一条断言（如：「我们会度量场景留存」「指标体系覆盖了所有核心问题」）。逐条做三层校验：

- **基础校验**：来源是否明确？口径是否一致？样本是否合理？
- **交叉推断**：同一主题下多个数据之间是否自洽？只用通识或公理做推论，不依赖材料外的信息：
  - 子集贡献上限：若新用户留存 +10%，但新用户仅占 5%，最大贡献整体留存 0.5%，与整体 +1% 矛盾
  - 数学公理：播放率 +5pt / +2% 意味着基础播放率为 250%，超过 100% 上限，不成立
  - 量级常识：某指标同比增长 200%，需判断基数是否过小，增长是否有实际意义
  - 方向一致性：多个相关指标同时呈现，方向是否互相支撑还是互相矛盾
- **跨断言一致性**：各条断言（含隐性断言）之间是否互相矛盾？「A 断言为真」是否与「B 断言为真」冲突？若存在矛盾，在相关断言处均标注「断言内部矛盾」，不单独开模块重复输出。

发现矛盾时，标注「数据内部不自洽」或「断言内部矛盾」，请用户核实，不继续基于此数据推论。

每条断言完成上述三层校验后，追加最后一问才能离开：这组数据给出的结论，是否是对这组数据唯一合理的解读？能给出替代解读且文档未排除时，标注「解读未封闭」。常见情况：占比上升可能是分子增长也可能是分母缩小；A 和 B 同时发生不等于 A 导致了 B；过去三个月的趋势不等于未来的趋势。

**Goodhart 审查**：完成所有断言校验后，对方案选定的核心成功指标逐条问：这个指标一旦成为 KPI，最容易的作弊路径是什么？该路径会损害真正的业务目标吗？发现 Goodhart 风险时作为隐性断言补入断言列表，标注「Goodhart 风险」。

### 价值分析维度

**收益价值分析**（独立模块，专门回答「值不值得做」）：

收益规模合理性：
- 方案声称的收益绝对值是否与项目级别、资源投入相称？
- 对比项目 level（S/A/B 级）对应的预期收益区间，判断声称的回报是否匹配
- 收益和投入是否在同一量级？若投入大但收益小，方案是否给出了其他（战略/防守/协同）收益维度

增长路径可信度：
- 梳理完整收益路径：触达 -> 转化 -> 留存 -> 变现（或 MAU -> 活跃 -> 付费），标出哪些节点有数据支撑、哪些是假设
- 路径中最薄弱的节点是哪个？如果这个节点失效，收益归零还是打折？

商业模式成立条件：
- 区分「我方可控」（自己开发、上线）和「他方履约」（合作方给流量、给分成）
- 他方履约类条件当前的确认状态：合同条款 / 协议框架 / 口头意向 / 未开始谈判
- 收益预测是否区分了乐观 / 基准 / 悲观场景？若未区分，给出三种场景的简单测算

**风险评估**：
- 用 Pre-mortem：假设一年后这个方案失败了，最可能的原因是什么？
- 用 Type 1/2：这个决策可逆吗？影响审查深度。
- Pre-mortem 识别出最高概率失败路径后，追问：方案里是否有对应的最小验证节点？如果假设不成立，最小损失的退出路径是什么？两者都缺失时，在结论中显式标注「缺少回退预案」。

### 材料优化维度

阅读时同步积累改进点，不需要在分析阶段单独执行。

**内容辅导**（修的是想法本身——逻辑、价值、解法、假设）：
- 假设链完整性：用假设树 + 最少驱动因子，找出真正独立的最小假设集，哪个需要补充论证？
- 差距定义清晰度：用 SCQA，问题框架有没有把冲突说清楚？
- 审查不只是找漏洞——如果方案里已有支撑某个结论的证据但作者没有显式论证出来，同样指出，这类「未被利用的有利支撑」和漏洞同等重要。
- 输出：内容层改进建议（逻辑漏洞 / 假设补充 / 论据加强），按影响优先级排序。

**表达辅导**（修的是传递噪声——内容正确但没有被读者正确接收到）：
- 结构合理性：用金字塔原理，结论是否前置？论据是否 MECE？
- 论点必要性：So What 测试，每个论点去掉后结论垮了吗？垮不了的可以删。
- 行动号召清晰度：读完这份方案，应该做什么决定——这一点是否明确？
- [可选] 受众适配扫描：不产生多版本材料，以注释形式输出两类标注：
  - **信息层**：不同受众对同一内容的理解偏好和关注点差异
  - **感受层**：材料中涉及特定受众的描述，是否可能引发对方的防御、误解或不适
  - 主流受众类型：技术/开发、BI/数据、业务/内容运营、垂直产品->平台产品、上级领导、跨部门决策者

以上内容辅导与表达辅导产生的改进点，**连同思路分析、事实分析、价值分析中所有诊断项里涉及「方案可改进之处」的内容**，统一汇总为「优化建议」，在评审结论中输出（见下方输出结构）。诊断项是各维度分析的就地标注，优化建议是对它们的摘要汇总视图——两者来源相同，不允许出现「详细分析有诊断但优化建议未收录」的断环情况。

## 输出结构

**分析完成后，按以下顺序一次性写出所有内容。各模块是同一份分析结果的不同视角切片，不代表分析的执行顺序。**

### 方案速览

#### 核心主张
一句话，作者最终希望你做什么判断或决定。

#### 骨架地图
带树形连接符的 ASCII 树结构，标注 [骨]/[肌]/[筋]。**必须使用 `├──` / `└──` / `│` 连接符，禁止用纯缩进空格替代。** 写入 POPO 时用 `<code-block>` 包裹。

#### 概念扫清
表格格式（写入 POPO 时用 `<table>` 标签）：

| 术语 | 释义 | 备注 |
|------|------|------|
| T+1 | 次日数据，属于滞后信号 | |
| XX项目 | 材料未做解释 | 需用户补充背景 |

### 评审结论

```
**诊断** 🟢/🟡/🔴 **[总体评判]** — 通过 / 条件通过 / 暂缓 / 否定

核心理由：
1. ...
2. ...

关键风险：...

资源建议：...
```

**关于「条件通过」**：不是固定模板，是自然涌现的结论——审定通过且改进建议非空时，自然形成「通过，但需改进 X」。

#### 优化建议

**来源**：思路分析、事实分析、价值分析中所有诊断项里涉及「方案可改进之处」的内容 + 材料优化维度（内容辅导 / 表达辅导）产生的改进点，两者合并、去重、统一排序后输出。

**规则**：
- 详细分析中每条🔴/🟡诊断，若指向「方案中存在问题或缺失」，必须在此收录，不允许仅出现在详细分析中而优化建议漏收
- 🟢诊断通常代表「通过」，不需收录；仅当确认为低优先级可改进项时收录为 P2
- **合并规则**：仅当「问题描述（What is broken）完全相同」且「行动方向完全重叠」时，才可将多条诊断合并为一条；若两条诊断的问题描述不同，即使建议方向相似，也必须分开列行。**🔴 诊断项不得因合并导致问题描述或优先级被降级**——若待合并的多条诊断中含有🔴，合并后必须保留所有🔴问题描述；若无法在同一行完整保留，应拆为独立行。位置/模块填最直接的来源章节。
- P0 对应 🔴，P1 对应 🟡，P2 对应 🟢，按优先级降序排列
- **类型**列填写该条建议来自哪个分析维度，从以下枚举中选择：
  - `问题定义`：SCQA 问题框架层面
  - `解法充分性`：假设树前提可信度（战略层或执行层）
  - `解法必要性`：干预点对齐、替代方案扫描
  - `事实分析`：断言校验、数据来源、口径、推导链、Goodhart 审查
  - `收益价值`：收益规模、增长路径、商业模式成立条件
  - `风险评估`：Pre-mortem 失败路径、回退预案、Type 1/2
  - `表达优化`：结构、金字塔原理、So What、行动号召（内容辅导/表达辅导来源）

### 详细分析

#### 思路分析

##### 问题定义
S/C/Q/A 四段，最后一行必须是诊断：
```
**诊断** 🟢/🟡/🔴 **[问题框架自洽性]** — 一句话说明
[说明冲突是否真实存在，问题定义是否准确]
```

##### 解法充分性
战略层假设树（最小独立集 + 最脆弱前提）；执行层假设树（仅逻辑链/结构性假设，数字参数不在此处展开）。

树形图中每个前提末尾只保留灯色 emoji，不展开文字；完整诊断集中放在树形图下方的诊断表：
```
+-- [前提1] ... 🟢
+-- [前提2] ... 🟡
    +-- [子前提2a] ... 🔴
```

树形图之后，紧接前提诊断表：
```
**诊断** 🟢 **[前提1可信度]** — 一句话说明
[展开解释]

**诊断** 🟡 **[前提2可信度]** — 一句话说明
[展开解释]

**诊断** 🔴 **[子前提2a可信度]** — 一句话说明
[展开解释]
```
如触发多方案对比，在此输出对比框架。

##### 解法必要性
三步结构：① 根因溯源（纵向逐层下钻，每层命名 + 层间找裂缝，直到触及不可再分的底层元素）；② 干预点对齐（根因层 / 症状层，症状层时显式标注），结束时输出诊断；③ 替代方案扫描（2-3 个替代路径及未选理由，材料未给出时标注「缺失项」），结束时输出诊断：
```
① 根因溯源：
  [层名1] 现象：...
           ↓ 裂缝：...（为何此层无法自我解释）
  [层名2] 机理：...
           ↓ 裂缝：...
  [层名N] 底层约束：...（触底标志：再往下只剩同义反复或人性/逻辑本身）

② 干预点对齐：...
**诊断** 🟢/🟡/🔴 **[干预点对齐]** — 必要性成立 / 症状层干预 / 缺失

③ 替代方案扫描：...
**诊断** 🟢/🟡/🔴 **[替代方案覆盖度]** — 对比框架完整 / 缺失对比 / 替代路径未分析
```

---

#### 事实分析

##### 关键断言校验（含隐性断言、跨断言一致性、Goodhart 审查）
包含显性断言（数字、明确主张）和隐性断言（承诺了但未显式陈述的条件、指标覆盖承诺等）。逐条格式：
```
**[断言 N]** "..."（隐性断言标注：[隐性]）
  基础校验：...
  交叉推断：...
  跨断言一致性：[与其他断言无矛盾 / 与断言X矛盾 → 标注：断言内部矛盾]
  **诊断** 🟢/🟡/🔴 **[解读封闭性]** — 唯一解读 / 解读未封闭（替代解读：...）
```

所有断言完成后，Goodhart 审查作为附加断言追加（对核心成功指标逐条）：
```
**[断言 N+k]** [隐性] "优化[指标名]等同于优化真正的业务目标"
  Goodhart 作弊路径：[最容易的指标作弊方式]
  **诊断** 🟢/🟡/🔴 **[Goodhart 风险]** — 指标健康 / 存在替代路径损害真正目标
```

---

#### 价值分析

##### 收益价值分析
收益规模合理性 / 增长路径可信度 / 商业模式成立条件，三个维度逐一输出。每个维度结束时输出诊断：
```
**诊断** 🟢/🟡/🔴 **[收益规模合理性]** — 一句话说明

（增长路径图之后）
**诊断** 🟡/🔴 **[最薄弱节点]** — 路径中最容易断的环节及影响
```

##### 风险评估
Pre-mortem 最高概率失败路径（列 2-3 条）；Type 1/2 判断；每条失败路径结束时输出诊断。**Pre-mortem 标头必须显式写出**，方便读者识别方法论来源：
```
**Pre-mortem 失败路径分析**
（假设 N 个月后方案失败，列最可能的 2-3 条原因）

**路径1（最高概率）：[失败模式名称]**
[描述]
**诊断** 🔴/🟡 **[回退预案]** — 有预案 / 缺少回退预案

**路径2（中等概率）：...**
...

**Type 1/2 判断**：[Type 1 不可逆 / Type 2 可逆] — [对审查深度的含义]
```

## 交互原则

**节奏**：全程不中断，除非遇到重大数据矛盾。在断言校验中发现严重数据问题时，主动标注并询问是否继续。

**措辞**：所有判断都标注依据，不单独说「有问题」。区分「事实」（材料里写的）和「判断」（我的分析）。允许不确定：「这个假设我无法从材料里验证，需要你补充」。

**低质材料**：逻辑混乱、数据缺失是常态。出现时：① 先做能做的部分；② 明确说明哪些地方因材料质量问题无法分析；③ 询问用户是否能补充背景信息。

### Step 2：同步至 POPO 在线文档

以 Step 1 的 MD 内容为唯一来源，逐节转换后写入 POPO。**内容必须与 MD 完全一致，不精简、不改写。**

写入 POPO 时特别注意：
- **来源链接**：MD 里的 `[标题](url)` → POPO 里用 `<a href="url">标题</a>` 标签，可正常渲染为可点击链接
- **概念扫清**：MD 里的列表 → POPO 里用 `<table>` 呈现（参考 POPO-IO-Guide.md 转换规则）
- **内容辅导 / 表达辅导**：MD 里的表格 → POPO 里用 `<table>` 呈现
- **假设树形图 / 骨架地图**：用 `<code-block>` 包裹；树形图后紧接诊断段落，不混入树内
- **多子项内容换行**：参考 POPO-IO-Guide.md 第 3 节「多子项内容的换行」，每个子项独立成一个 `<p>`，禁止用 `<br/>` 拼接

写入 POPO **必须** 用 `popo_io.py` Python API，**禁止** 直接调 `python3 popo_io.py` 的 CLI 方式：
```python
# popo_io.py 位于本 Skill 的 scripts/ 子目录，用 importlib 按绝对路径加载
import importlib.util
spec = importlib.util.spec_from_file_location(
    "popo_io", "<SKILL_DIR>/scripts/popo_io.py"
)
pio = importlib.util.module_from_spec(spec); spec.loader.exec_module(pio)
doc_id = pio.create_doc('文档标题')
pio.write_sections(doc_id, [...])   # 传 list[str]，每项为合法 HTML 字符串
print(pio.doc_url(doc_id))
```
其中 `<SKILL_DIR>` 为本 Skill 所在目录的绝对路径，Claude 运行时根据 SKILL.md 位置自行替换。

格式规范、转换规则、已知坑、代码模板，全部参考：
**`references/POPO-IO-Guide.md`**
